# Build Process — Kratom News Today

This document describes how the site builds and deploys. Useful for understanding what GitHub Actions does, for debugging build issues, and for any agent maintaining the build pipeline.

## Stack overview

- **Static site generator:** Eleventy v3
- **Template language:** Nunjucks
- **CSS:** Plain CSS (no preprocessor), organized into files
- **JavaScript:** Vanilla JS, no framework
- **Markdown:** standard markdown via `markdown-it` (Eleventy default)
- **YAML frontmatter:** parsed by Eleventy automatically
- **Image processing:** Sharp via Eleventy plugin (for OG image generation)
- **RSS:** `@11ty/eleventy-plugin-rss`
- **Sitemaps:** generated by template files, no plugin needed

## Build sequence

When `npx @11ty/eleventy` runs:

1. **Read frontmatter and content** from `./content/briefings/` and other source markdown files
2. **Build collections** (briefings, briefingsByBeat, opinions if any exist)
3. **Resolve internal links** — the slug-reference markdown plugin processes `[link text][slug]` patterns using `related_briefings` arrays in frontmatter
4. **Render templates** — each page template renders against its data
5. **Generate sitemaps** — main sitemap and news sitemap from collection data
6. **Generate RSS feeds** — full feed and beat-specific feeds
7. **Generate search index** — `scripts/build-search-index.js` produces `_site/search-index.json` from all briefings
8. **Process CSS and JS** — copied to `_site/assets/`
9. **Copy OG images** — from `src/assets/og-images/` to `_site/assets/og-images/`
10. **Write output** to `_site/`

Build typically completes in 5-30 seconds depending on archive size. Year-1 archive (~365 briefings) should still build in under a minute.

## GitHub Actions deploy workflow

`.github/workflows/deploy.yml`:

Triggered on push to `main`. Steps:

1. Checkout the repo
2. Set up Node.js (v20 LTS)
3. Install dependencies via `npm ci`
4. Run validation script (verifies frontmatter, link resolution, required fields)
5. Run Eleventy build (`npx @11ty/eleventy`)
6. Set up SSH using the secret private key
7. Rsync `_site/` contents to Cloudways server via SSH
8. Notify on failure (logs in GitHub Actions output)

The workflow uses these secrets:
- `CLOUDWAYS_SSH_PRIVATE_KEY`
- `CLOUDWAYS_SSH_HOST`
- `CLOUDWAYS_SSH_USER`
- `CLOUDWAYS_DEPLOY_PATH`

Typical workflow run time: 60-90 seconds total. Push to main → site live in under 2 minutes most of the time.

## OG image generation

OG images are generated per-article using a Node.js script (`scripts/generate-og-image.js`) that runs at publish time (called from the publishing workflow on Openclaw), NOT during the Eleventy build.

Reason: Eleventy builds run on every push, including changes to templates. Regenerating OG images on every build is wasteful. Generating them once at publish time (when a briefing is created) and committing them to the repo makes builds fast and idempotent.

The script:
1. Takes a slug, title, beat, and date as arguments
2. Uses Sharp to render an SVG template with the article data
3. Outputs a 1200x630 PNG to `src/assets/og-images/<slug>.png`

The SVG template uses the same colors and typography as the site. The publishing workflow commits the generated image alongside the article markdown.

If the script fails for any reason, the article publishes with a default fallback OG image at `src/assets/default-og.png`. The fallback is the publication wordmark on the brand background. Not as good as per-article images but acceptable.

## Validation script

`scripts/validate.js` runs as part of the GitHub Actions workflow before Eleventy builds. Checks:

- All briefing markdown files have valid YAML frontmatter
- All required frontmatter fields are present
- All `related_briefings` slugs resolve to either existing files or to slugs in the recent past
- All `sources` entries have required fields
- No briefing has an empty body
- All inline reference-style links have their slug in `related_briefings`

If validation fails, the build halts and the deploy is skipped. Failure is logged with specifics.

## Local development

For local development on the site infrastructure (not on content):

```
npm install
npx @11ty/eleventy --serve
```

This runs Eleventy in watch mode at `http://localhost:8080`. Changes to templates, components, CSS, JS auto-rebuild and refresh the browser.

For testing content changes locally:
- Add or modify markdown files in `./content/briefings/`
- The local server picks them up automatically
- Verify rendering, schema, internal links, etc. before pushing

## Dependencies overview

Key npm dependencies (full list in package.json):

- `@11ty/eleventy` — static site generator
- `@11ty/eleventy-plugin-rss` — RSS feed generation
- `markdown-it` — markdown parsing
- `markdown-it-attrs` — attribute support in markdown
- `sharp` — image processing for OG images
- `gray-matter` — frontmatter parsing (used by validation and search index scripts)
- Plus development tools (no test framework at launch; can add later if needed)

## Updating dependencies

When dependencies need updating:
1. Run `npm outdated` to see what's behind
2. Update conservatively — patch versions safe, minor versions usually safe, major versions need testing
3. Run `npm install` then `npx @11ty/eleventy` locally to verify build still works
4. Commit `package.json` and `package-lock.json` together
5. Push and verify GitHub Actions still completes

Avoid updating dependencies on the same commit as content changes — separates concerns and makes rollback easier if needed.
